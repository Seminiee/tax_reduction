# UI_SPEC.md — 프론트엔드 UI/UX 명세

> **Stage 21 안내**: 이 문서의 원안(투자금액 슬라이더 + 절세구간 마커)은 더 이상 `/`에 사용되지 않는다. `/`는 Stage 21부터 매매차익 UI로 통합되었으며 최신 구조는 `design/merged-hold-trade-mockup.html`과 `skills.md` 8절을 따른다. 원안은 기록 목적으로 보존한다.

Stage 3(frontend-ui) 작업 시 이 문서와 `design/ui-mockup.html`을 1차 레퍼런스로 삼는다. `design/ui-mockup.html`은 정적 시안이 아니라 실제 동작하는 vanilla JS 목업이므로, 브라우저로 직접 열어서 상호작용을 확인한 뒤 React 컴포넌트로 옮긴다.

## 1. 디자인 언어
- 레이아웃: 라벨(왼쪽, 회색 배경 박스) + 입력/선택 영역(오른쪽) 구조의 카드형 행. 행 사이는 구분선, 카드 전체는 둥근 테두리(`border-radius: 16px`) + 얇은 보더.
- 컬러
  - accent(주 강조): `#2f5fe0`
  - accent-soft(강조 배경): `#eef2ff`
  - safe(비과세 구간): `#1fa971`
  - danger(초과세 구간 / 혜택취소 경고): `#e05a2f`
  - border: `#e6e8ec`, label 배경: `#f4f5f7`
- 헤더 구성: 큰 제목 → 파란색 키워드 서브타이틀 → 회색 설명 서브타이틀 → `updated at` 캡션(세율 기준일 명시, 항상 노출)
- 폰트: 시스템 sans-serif (`-apple-system, "Noto Sans KR"` 등), 별도 웹폰트 로드 없이 구현

## 2. 컴포넌트 구성 (위→아래 순서)
1. **헤더**: 제목 + 서브타이틀 2줄 + updated 캡션
2. **조건 입력 카드**
   - ISA 유형 2단 토글 (일반형 200만원 / 서민형 400만원) — 선택값이 비과세 한도에 즉시 반영
   - 예상 연수익률 숫자 입력 (%)
   - 보유기간 숫자 입력 (년)
3. **투자금액 슬라이더 카드** (핵심 기능)
   - 상단에 현재 금액 실시간 표시 (큰 폰트, accent 컬러)
   - 슬라이더 트랙: 비과세 구간(safe)과 초과세 구간(danger)을 트랙 위에서 색으로 분리
   - **마커**: 트랙 위 특정 지점에 세로선 + 라벨로 "여기부터 초과세" 표시
   - 트랙 하단 최소/최대 범위 라벨
   - 슬라이더 아래 안내 문구(note) — 상태에 따라 3가지 톤으로 분기 (아래 4절)
4. **결과 비교 카드**: 일반계좌 vs ISA 계좌 세후 순자산 2단 박스
   - 세후 순자산 금액(큰 폰트) + 세금 상세(작은 폰트, 어떤 규칙으로 계산됐는지 한 줄 설명)
   - 더 유리한 쪽에 accent 테두리 + "이 조건에서 더 유리" 배지 (절대 "무조건 유리"로 쓰지 않음 — skills.md 5절 조건부 표현 규칙과 동일)
5. **하단 disclaimer**: 실제 계산은 `lib/tax` 함수를 따른다는 문구 + 세율 미검증 안내, 항상 노출

## 3. 마커 위치 계산 로직 (Stage 1 lib/tax와 연동 필수)
슬라이더는 "투자금액"이지만 비과세 한도는 "실현 수익" 기준이므로, 투자금액을 그대로 한도와 비교하면 안 된다. 마커 위치는 다음을 만족하는 투자금액을 역산한 지점이다:

```
growthFactor = (1 + 연수익률)^보유기간 - 1
threshold_투자금액 = ISA_비과세한도 / growthFactor
```

Stage 3 구현 시 이 역산 로직은 `lib/tax`에 순수함수로 추가하고(예: `findIsaThresholdPrincipal`), 컴포넌트에서 재계산하지 않는다. (CLAUDE.md 규칙: 세금 로직과 UI 로직 분리)

## 4. 안내 문구(note) 상태 분기 — 4가지
1. **정상 구간 내**: "투자금액이 {threshold}만원을 넘으면 예상 수익이 비과세 한도({limit}만원)를 초과해요. 초과분부터는 9.9% 분리과세가 붙습니다."
2. **슬라이더 전 구간이 이미 초과**: threshold가 슬라이더 최소값보다 작을 때 — "슬라이더 최소 금액에서도 이미 비과세 한도를 초과해요. 전액이 9.9% 분리과세 대상입니다." (트랙 전체 danger 색)
3. **슬라이더 전 구간이 안전**: threshold가 슬라이더 최대값보다 클 때 — "이 조건에서는 슬라이더 전 구간이 비과세 한도 이내예요." (트랙 전체 safe 색)
4. **보유기간 3년 미만**: 마커/구간 표시를 아예 숨기고 "보유기간이 3년 미만이면 ISA 세제 혜택이 전부 취소돼요." 경고(danger)로 대체

## 5. 접근성/카피 원칙
- 모든 절세 관련 문구는 조건부 표현 사용 ("이 조건에서는~", "~일 때") — 확정적 단정 금지
- 세율/한도 수치가 바뀔 수 있다는 disclaimer는 항상 화면에 노출 (스크롤로 사라져도 안 됨 — footer 고정 또는 상시 표시 영역 권장)

## 6. Stage 5: 세금 Q&A 챗봇 (`ChatPanel`, 구현 완료 — 실제 구현 기준 기록)

이 섹션은 mockup에는 없던 Stage 5(선택 기능) 산출물이다. UI_SPEC.md 1절 디자인 언어(색상 토큰·카드 스타일)를 그대로 따르되, 컴포넌트 자체는 사후에 결정한 것이라 실제 구현 기준으로 문서화한다.

- **위치**: 결과 비교 카드 → "AI 설명 보기" 패널 다음, disclaimer 앞. 별도 `.card`(border-radius 16px, 흰 배경)로 다른 카드와 동일한 리듬을 유지한다.
- **접힘/펼침**: 기본 상태는 접힘. 카드 전체 너비의 토글 버튼(`좌: "세금 Q&A 챗봇에게 물어보기"`, `우: "펼치기 ▼" / "접기 ▲"`)을 누르면 펼쳐진다. 배경/보더 변경 없이 텍스트만 바뀌는 방식으로 최소한의 애니메이션 비용을 유지한다.
- **펼친 상태 내부 구성 (위→아래)**:
  1. **상시 disclaimer**: `label-bg`(#f4f5f7) 배경의 작은 박스에 "AI 답변은 참고용이며 개별 세무 상담을 대체하지 않습니다." — 대화 내용과 무관하게 챗봇이 열려 있는 한 항상 최상단에 고정 노출(스크롤되어도 사라지지 않음).
  2. **대화 없음 힌트**: 첫 진입 시 예시 질문 2개를 회색 텍스트로 노출("ISA 비과세 한도가 얼마예요?" 등) — 빈 화면으로 남기지 않기 위함.
  3. **메시지 리스트**: 최대 높이 320px, 넘치면 세로 스크롤. 사용자 말풍선은 우측 정렬 + `accent`(#2f5fe0) 배경 + 흰 글씨 + 말꼬리가 우하단에 오도록 `border-radius: 12px 12px 2px 12px`. AI 말풍선은 좌측 정렬 + `label-bg` 배경 + 기본 텍스트색 + 말꼬리가 좌하단(`12px 12px 12px 2px`). 로딩 중에는 AI 말풍선 자리에 "답변 작성 중..." 임시 표시.
  4. **입력 행**: 텍스트 입력(1000자 제한, Enter로 전송) + "전송" 버튼(`primaryButton` 스타일 재사용, accent 배경). 에러 발생 시 기존 `.note.noteDanger`(주황 배경) 컴포넌트를 그대로 재사용해 톤을 통일한다.
- **상태 관리**: 완전히 클라이언트 상태(React state)로만 관리하며 서버에 대화가 저장되지 않는다(stateless API). 페이지를 새로고침하면 대화가 초기화된다.
- **시뮬레이션 맥락 연동**: 사용자가 화면 위쪽에서 조정한 조건(투자금액/수익률/보유기간/ISA유형)과 그 계산 결과가 매 전송마다 함께 서버로 전달되어, AI가 "당신이 방금 계산한 조건 기준으로는~"처럼 답할 수 있다. 이 값은 슬라이더를 움직여도 챗봇을 다시 열 필요 없이 항상 최신 값으로 갱신된다.
- **디자인 토큰 재사용**: 새로 정의한 색상은 없다 — accent/accent-soft/label-bg/border/text/text-sub/danger를 그대로 재사용해 다른 카드들과 시각적으로 이질감이 없도록 했다.

## 7. Stage 7 갱신: 챗봇 공용화 (6절의 위치/색상 설명을 대체)

Stage 7에서 이 서비스가 `/`(거치 기준 수익) 하나에서 `/`, `/trade`(매매차익, Stage 8 예정) 다중 도구 구조로 확장되면서 챗봇(`ChatPanel`)이 `components/tax-simulator/`에서 `app/layout.tsx` 레벨의 공용 컴포넌트(`components/chat/`)로 옮겨졌다. 6절에서 설명한 **메시지/disclaimer/입력 동작 원칙은 그대로 유효**하지만, 아래 두 가지는 6절 내용을 대체한다.

- **위치**: 더 이상 결과 비교 카드 아래에 inline으로 붙지 않는다. 화면 우하단 고정(`position: fixed`) 플로팅 버튼("세금 Q&A 챗봇")으로 존재하며, 클릭하면 그 위에 패널이 펼쳐진다. (Stage 14에서 이 위치/형태가 다시 바뀐다 — 8절 참고.)
- **색상**: 6절에 설명된 accent(블루)/label-bg 등 `/` 페이지 전용 토큰 대신, 두 도구 색과 무관한 슬레이트/네이비 중립색을 새로 쓴다 — 배경 `#0f172a`(slate-900), 강조/사용자 말풍선 `#1e293b`(slate-800), 옅은 배경 `#f1f5f9`(slate-100), 보더 `#e2e8f0`(slate-200). 정의는 `components/chat/ChatPanel.module.css` 참고. 이 색상 원칙은 Stage 14 이후에도 그대로 유효하다.

`ChatContext`(`components/chat/ChatContext.tsx`)가 대화 히스토리와 "현재 시뮬레이션 조건"을 들고 있어, 페이지를 이동해도(`/` ↔ `/trade` ↔ `/dividend`) 대화가 초기화되지 않는다.

## 8. Stage 14 갱신: 챗봇을 하단 고정 바로 재설계 (7절의 위치 설명을 대체)

우하단 원형 버튼("세금 Q&A 챗봇")을 눌러야 패널이 뜨는 구조가 발견성이 낮다는 문제로, 화면 하단에 항상 고정된 가로 바 형태로 바꿨다. 7절에서 설명한 색상/disclaimer/버블 스타일 원칙은 그대로 유효하다.

- **기본(collapsed) 상태**: 뷰포트 하단 전체 폭에 고정된 얇은 입력 바(높이 약 56~64px) — 아이콘 + "세금 관련 질문을 입력하세요" placeholder 입력창 + 전송 버튼. 별도로 열어야 하는 버튼이 없다.
- **펼침(expanded) 상태**: 입력창에 포커스하거나 메시지를 보내면 입력 바 위로 대화 히스토리 영역이 펼쳐진다(`max-height: 50vh`, 넘치면 스크롤). 헤더의 "접기 ▾" 버튼으로 다시 collapsed 상태로 되돌릴 수 있으며, 이때도 대화 내용(`ChatContext`)은 그대로 남는다 — 접었다 펼쳐도 히스토리가 사라지지 않는다.
- **페이지 하단 여백**: `app/layout.tsx`가 `{children}`을 감싸는 wrapper에 `pb-20`(80px)을 둬서 collapsed 바가 페이지 콘텐츠(특히 하단 disclaimer)를 가리지 않는다.
- 정의는 `components/chat/ChatPanel.tsx`, `components/chat/ChatPanel.module.css` 참고.

## 8. Stage 21: 거치식/매매차익 통합

`/`가 거치식(TaxSimulator) 단독 도구에서 매매차익 계산기 UI로 통합되면서, `design/merged-hold-trade-mockup.html`(사용자가 직접 확정한 최종 시안)이 `/`의 1차 레퍼런스가 된다. 페이지는 위→아래로 3개 영역으로 구성된다.

1. **헤더(A)**: 제목 "절세 계좌 수익 시뮬레이터". 그 아래 서브타이틀은 기존 거치식 헤더(`components/tax-simulator/Header.tsx`)의 문구를 최대한 그대로 가져오되 색상만 매매차익 테마(에메랄드, `text-emerald-600`)로 통일한다. `updated at` 캡션도 그대로 유지한다(`config/tax-rules.json`의 `as_of` 참조).
2. **본문(B)**: 기존 `/trade`의 UI를 그대로 사용한다 — AI 자연어 입력(매매차익 스키마: 종목명/현재주가/주당손익/수량) → ISA 유형 토글 → 투자 시나리오 설정 카드 → 수량 슬라이더 → 결과 카드(헤드라인 절세액 + 납입한도 게이지 + 비과세소진율 게이지 + 세금비교 바 2개).
3. **AI 설명 보기 (신규)**: 페이지 맨 밑, 전체 폭 섹션. 버튼을 누르면 아래로 펼쳐지며 AI가 생성한 결과 해설 텍스트를 보여준다(Stage 4의 기존 "AI 설명 보기" UX 패턴 재사용 — 접혔다 펼쳐지는 형태, 최초 클릭 시에만 API를 호출하고 이후에는 캐시된 결과를 그대로 보여준다). `components/trade-calculator/AiExplanationPanel.tsx` 참고.

**라우팅**: `/trade`는 `/`로 리다이렉트된다(`next/navigation`의 `redirect()`). 상단 네비게이션(`components/site-shell/SiteNav.tsx`)은 "절세 계좌 수익 시뮬레이터"(`/`), "배당금 계산기"(`/dividend`) 2개 링크로 줄어든다.

**계산 로직**: `/`의 실제 계산은 `lib/tax/trade-calculator.ts`의 `calculateTrade`를 그대로 호출한다(중복 구현 없음). 거치식 계산 엔진(`lib/tax/general-account.ts`, `isa-account.ts`, `threshold.ts`)과 `TaxSimulator.tsx`, `/api/simulate`, `/api/parse`(거치식 파싱)는 삭제하지 않고 그대로 남겨둔다 — 더 이상 어떤 페이지에서도 호출되지 않는 죽은 코드가 되지만, 나중에 되살릴 수 있게 보존한다(Stage 18의 `otherFinancialIncomeKrw` 보존 원칙과 동일). `/api/simulate`와 `/api/parse`는 여전히 살아있는 엔드포인트이므로 보안 점검 체크리스트에서는 계속 포함된다.

**AI 설명 보기 API**: `/api/explain`의 요청 스키마가 `ChatCurrentSimulation`과 같은 `kind` 판별 유니언(`{ kind: "hold", ... } | { kind: "trade", ... }`)으로 일반화되었다. `/` 페이지는 `kind: "trade"`로 호출한다. 원문은 `PROMPTS.md` 2절 v2 참고.
